# Community growth and evidence plan

The goal is verifiable adoption of public market-data infrastructure, not vanity metrics or trading-performance claims.

## Contributor path

1. Use the issue forms to report a reproducible bug, request a bounded feature, propose a public data source, or document a verifiable downstream use case.
2. Select a small task from [GOOD_FIRST_ISSUES.md](GOOD_FIRST_ISSUES.md).
3. Follow [CONTRIBUTING.md](../CONTRIBUTING.md) and include mocked tests for connector changes.
4. Submit a pull request using the repository template.
5. Record validation commands and preserve the no-signal, no-execution boundary.

## Adoption evidence

Evidence is useful only when another person or project can be verified. Suitable evidence includes:

- a public downstream repository that imports the package or consumes its artifacts;
- an external issue describing a concrete workflow;
- an external pull request;
- an independent technical post or tutorial;
- a public package reference; or
- a maintainer-approved testimonial with permission to quote it.

Raw downloads are operational signals, not proof of unique users, organizations, production use, or downstream dependence.

## Claude for Open Source

The official Maintainer Track threshold and discretionary Ecosystem Impact Track are documented in [CLAUDE_FOR_OSS_READINESS.md](CLAUDE_FOR_OSS_READINESS.md). Community work should improve real usefulness and produce verifiable evidence; it must not use purchased stars, automated engagement, reciprocal star schemes, or misleading adoption claims.
